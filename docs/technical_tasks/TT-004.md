# Technical Task: Auto-Verify Skill Inventory Tables

## Overview

| | |
|---|---|
| **Task ID** | TT-004 |
| **Task Name** | Auto-Verify Skill Inventory Tables |
| **Category** | Infrastructure |
| **Goal** | The skill inventory (directory tree / Quick Reference tables) is hand-duplicated in `README.md`, `CLAUDE.md`, and `nexa-skills/SKILL.md`. `CONTRIBUTING.md` already documents this as a manual sync obligation. Add an automated check — mirroring `scripts/sync-shared.sh`'s conventions — that fails CI when a skill is added, renamed, or removed without updating all three listings. |
| **Status** | Done |

## Acceptance Criteria

- [x] A script (`scripts/check-skill-inventory.sh`) detects drift between actual skill directories and the three hand-maintained listings
- [x] The check also flags a skill whose `name:` frontmatter doesn't match its directory name
- [x] The check runs in CI (new workflow) so a missed listing update fails the PR
- [x] `CONTRIBUTING.md`'s "When you add, rename, or remove a skill" section references the automated check

## Affected Areas

- `scripts/check-skill-inventory.sh` (new)
- `.github/workflows/check-skill-inventory.yml` (new)
- `CONTRIBUTING.md`

## Dependencies

- TT-001 (run last as a final regression check that the SKILL.md split didn't corrupt any skill's `name:`/frontmatter invariant)

## Decisions

| Decision | Provenance | Source/Reasoning |
|----------|------------|------------------|
| Verify presence/absence, not regenerate the 3 tables | INFERRED | The 3 listings use genuinely different formats and independently-worded descriptions; faithfully regenerating all 3 is a bigger, riskier lift than the issue calls for. |
| New workflow file, not extending `sync-shared.yml` | EXPLICIT | `sync-shared.yml`'s path filters don't cover `nexa-claude-core/skills/**`, `README.md`, or `CLAUDE.md` — the exact files this check targets — and conflating two differently-triggered scripts into one job would muddy `sync-shared.yml`'s single existing responsibility. |
| `nexa-skills` is excluded from the "must be listed as `/skill-name`" check, though still validated for frontmatter `name:` match | INFERRED | The orchestrator skill doesn't list itself as a slash command in its own Quick Reference table, nor in README.md/CLAUDE.md (verified: no `` `/nexa-skills` `` reference exists anywhere in the repo today). Flagging this as drift would be a false positive against intentional, existing behavior. |
| Reverse "phantom" check (listed `/skill-name` with no matching directory) extracts every backtick-wrapped `` `/word` `` token from the 3 files rather than parsing table rows structurally | INFERRED | Verified empirically that every such token in README.md/CLAUDE.md/nexa-skills/SKILL.md today corresponds 1:1 to an actual skill directory (32 matches, no false positives) — the ASCII directory trees and prose/installation snippets in these files don't use single-backtick `/word` formatting, so this simpler extraction is safe today. Accepted as a known fragility (see code-review note below), not a defect. |
| Added `\|\| true` to the two grep pipelines that build `name` (frontmatter) and `listed` (phantom check) — corrected via code review | INFERRED (corrected via code review) | Under `set -euo pipefail`, a `grep` with zero matches (e.g. a `SKILL.md` with malformed/missing `name:` frontmatter) returns exit 1 and aborts the whole script silently via `set -e`, defeating the check's purpose of failing loudly. `scripts/sync-shared.sh:31` already guards its own equivalent pipeline with `\|\| true` — this mirrors that convention. Verified with a malformed-frontmatter test case: before the fix the script produced no output; after, it correctly reports `MISMATCH ... frontmatter name '' != directory '...'` and exits 1. |
| Script validated only against pre-TT-001 file structure | EXPLICIT (process note) | TT-001 (splitting oversized SKILL.md files into SKILL.md + REFERENCE.md) is being implemented in parallel and wasn't merged when this was built/verified. TT-001 only splits SKILL.md content into a same-named SKILL.md + a new REFERENCE.md in the same directory — it does not touch skill directory names, `name:` frontmatter, or the 3 listing files — so this check should not need any code change once TT-001 lands. Re-run `scripts/check-skill-inventory.sh --check` after TT-001 merges as a real regression check, per TT-004's own dependency note. |
