# Technical Task: Split Oversized SKILL.md Files

## Overview

| | |
|---|---|
| **Task ID** | TT-001 |
| **Task Name** | Split Oversized SKILL.md Files |
| **Category** | Cleanup |
| **Goal** | Anthropic's skill-authoring guidance caps `SKILL.md` bodies at ~500 lines with progressive disclosure (detail extracted to a linked reference file, loaded on demand). 7 skills exceed this by up to 2x, paying full token cost on every invocation regardless of which section is relevant. Split each into a lean `SKILL.md` (workflow skeleton) plus a `REFERENCE.md` (templates, detailed examples, edge-case tables), mirroring the existing `nexa-claude-core/skills/requirements/` pattern. |
| **Status** | Done |

## Acceptance Criteria

- [x] Every `SKILL.md` body listed in Affected Areas is under ~500 lines
- [x] Extracted content lives in a `REFERENCE.md` referenced directly from `SKILL.md` (one level deep, no nested references)
- [x] Headings referenced by name elsewhere (in-file or from another file, e.g. `agents/playwright-test.md`) are kept as stubs in `SKILL.md`, never deleted
- [x] `CONTRIBUTING.md`'s "Anatomy of a skill" section documents the reference-file pattern and size guidance
- [x] Each refactored skill is manually re-invoked (or spot-checked) to confirm behavior is unchanged

## Decisions

| Decision | Provenance | Source/Reasoning |
|----------|------------|------------------|
| Single `REFERENCE.md` per skill directory, not multiple named files | INFERRED | Mirrors the one existing precedent (`requirements/SKILL.md` + `REFERENCE.md`) exactly; avoids a per-skill naming decision made 7 times inconsistently. |
| Staged execution: pilot on `sprint-complete` + `playwright-test` first, then the remaining 5 | INFERRED | De-risks the recipe on the simplest case and the one case with pre-existing asset dirs before applying it to the largest/highest-stakes files. |
| Existing `templates/`/`examples/` asset dirs are referenced from `REFERENCE.md`, not duplicated into it | EXPLICIT | Avoids creating two sources of truth for the same code samples. |
| Verification method for the "manually re-invoked (or spot-checked)" AC: read every `SKILL.md` + `REFERENCE.md` pair in full plus a script-based anchor/dangling-reference check, rather than a live interactive re-run of each multi-phase skill | INFERRED | The AC's own parenthetical allows spot-checking as an alternative to live re-invocation; these are documentation-only, multi-phase interactive skills (some spanning 9 pipeline phases) that cannot be meaningfully re-run end-to-end in this session without a real target project and hours of interactive steps. Full-text read-back plus automated anchor verification directly tests the acceptance criterion that matters here (no content lost, no dangling links) without the impractical cost of a live run. |
| `playwright-test/SKILL.md`'s reference to `REFERENCE.md#env-e2e-contents` corrected to `REFERENCE.md#enve2e-contents` | EXPLICIT | GitHub's heading-slug algorithm strips punctuation (including the dots in `` `.env.e2e` Contents ``) without inserting a hyphen in its place, so the real generated anchor is `enve2e-contents`, not `env-e2e-contents`. Caught by a script that recomputes every heading's slug and diffs it against every `REFERENCE.md#...` link; this was a genuine dangling link that would have sent a reader to the top of the file. |
| Self-review (via `/code-review`) returned no Critical or Important findings — only two Minor observations (sprint-prepare/engineer-requirements have little line-count headroom left; the Nexa Rules Gate doesn't apply to this marketplace repo itself) | EXPLICIT | Both are informational, not defects, and require no code change; recorded here rather than acted on per the "skip Minor findings" guidance. |
