# Technical Task: Evaluation Harness for Skills

## Overview

| | |
|---|---|
| **Task ID** | TT-002 |
| **Task Name** | Evaluation Harness for Skills |
| **Category** | DevEx |
| **Goal** | No regression testing exists for skill/prompt behavior — `scripts/sync-shared.sh` only checks file-identity drift, not whether a skill still behaves correctly after an edit. Add scenario-based evaluations (per Anthropic's evaluation-driven-development guidance) for the highest-risk orchestrator skills, plus a documented process for running them, so skill edits can be validated repeatably. |
| **Status** | Done |

## Acceptance Criteria

- [x] At least 3 evaluation scenarios exist for each of: `deliver-use-case`, `resolve-bug`, `sprint-deliver`, `sprint-kickoff`, `sprint-complete`
- [x] Each skill's scenarios cover a happy path, a fix-loop/iteration path, and a failure-recovery/short-circuit path
- [x] Scenarios are stored in-repo as structured files (`evals/<skill-name>/EVAL-00N.json`), not just described in a PR description
- [x] `CONTRIBUTING.md` documents how to add and run evaluations for a new or edited orchestrator skill
- [x] A `RESULTS.md` log format exists per skill for recording PASS/FAIL runs over time

## Affected Areas

- `evals/deliver-use-case/`, `evals/resolve-bug/`, `evals/sprint-deliver/`, `evals/sprint-kickoff/`, `evals/sprint-complete/` (new)
- `CONTRIBUTING.md`

## Dependencies

- None

## Decisions

| Decision | Provenance | Source/Reasoning |
|----------|------------|------------------|
| Root-level `evals/<skill-name>/`, not per-plugin | INFERRED | The eval mechanism is generic, not nextjs-specific, even though this batch's 5 target skills happen to live in nexa-claude-nextjs; mirrors this repo's existing cross-plugin tooling convention (`scripts/` at root). |
| Runner is a documented manual/agent-assisted checklist, not a script | EXPLICIT | No CI compute exists to run real Claude Code sessions; Anthropic's own guidance states there's no built-in eval runner and teams build their own process. |
| Committed fixture target project (docs/requirements.md, use cases, Prisma schema, etc.) is out of scope for this TT | INFERRED | A much larger, separate lift than this issue calls for; preconditions are documented in prose and the evaluator arranges a scratch project at run time. |
| Added an explicit Code Review Gate step to `deliver-use-case` EVAL-001/EVAL-002 and `resolve-bug` EVAL-001/EVAL-002 (between DoD/DoD_BUG and Post-Implementation Tracking) | INFERRED (corrected via code review) | The reviewer found that both skills' `SKILL.md` files (as they exist in the shared repo, ahead of this worktree's branch point) define a `code-review`-subagent gate with its own 2-iteration cap, sitting between the Definition of Done check and Post-Implementation Tracking, with its own row in the terminal summary table and Failure Recovery text. The original scenarios omitted it, so a regression that silently bypassed that gate would still have passed the eval as written. `EVAL-003` for both skills was already unaffected since those scenarios fail before the Code Review Gate would run. |
| Reordered `resolve-bug` EVAL-001's Nexa Rules Gate / Sprint Branch Gate step to precede Step 0 | INFERRED (corrected via code review) | Minor ordering fix — the gates are declared in `resolve-bug/SKILL.md` before `## Pipeline`/Step 0, so the eval's `expected_steps` should reflect that order rather than listing them after Step 0's DoR_BUG check. |
