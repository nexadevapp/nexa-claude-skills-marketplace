# Use Case: Shorten a URL

## Overview

| | |
|---|---|
| **Use Case ID** | UC-002 |
| **Use Case Name** | Shorten a URL |
| **Primary Actor** | Anonymous Visitor |
| **Secondary Actor** | Link Owner (if authenticated) |
| **Goal** | Convert a destination URL into a short link that can be shared |
| **Status** | Approved |
| **Functional Requirements** | FR-001, FR-002 |

## Preconditions

- The shortener service is reachable.
- The visitor has a destination URL they want to shorten.
- The visitor's IP has created fewer than 20 links in the past hour (NFR-004).

## Main Success Scenario

1. Visitor opens the home page (`/`).
2. System displays the shortener form.
3. Visitor pastes a destination URL into the input field.
4. Visitor optionally expands the "Custom slug" panel and provides a slug (3–32 chars matching `^[a-z0-9-]{3,32}$`).
5. Visitor submits the form.
6. System validates the destination URL syntactically (RFC 3986; scheme must be `http` or `https`).
7. System checks the destination against the abuse blocklist (out of scope to manage in this release; the check itself runs).
8. System generates a unique 7-character slug using a URL-safe alphabet if no custom slug was provided.
9. System persists a `LINK` row with `slug`, `destinationUrl`, `creatorIp`, `ownerId` (the authenticated user's id, or `null`), and `expiresAt` (`createdAt + 30 days` if anonymous, otherwise `null`).
10. System displays the shortened URL in a result block and copies it to the visitor's clipboard.

## Alternative Flows

### A1: Invalid destination URL

**Trigger:** validation in step 6 fails (malformed, non-http/https, or empty).
**Flow:**

1. System shows an inline validation error under the URL field.
2. Visitor corrects the URL.
3. Use case continues at step 5.

### A2: Destination on the abuse blocklist

**Trigger:** the destination matches an entry in the blocklist (step 7).
**Flow:**

1. System rejects the request with a generic message: "This URL cannot be shortened."
2. System logs the attempt with the visitor's IP and the destination, so moderators can review patterns.
3. Use case ends in failure.

### A3: Custom slug already taken

**Trigger:** in step 8, a custom slug from step 4 collides with an existing `LINK.slug`.
**Flow:**

1. System shows the error "Slug unavailable" next to the slug field.
2. Visitor edits the slug or removes it.
3. Use case continues at step 5.

### A4: Generated slug collision

**Trigger:** in step 8, the randomly generated slug collides with an existing `LINK.slug`.
**Flow:**

1. System retries slug generation up to 5 times.
2. If still colliding, System returns HTTP 503 with a "Try again in a moment" message.
3. Use case ends in failure.

### A5: Rate limit exceeded

**Trigger:** the visitor's IP has already created 20 links in the past hour (precondition violated at submit time).
**Flow:**

1. System rejects the request with HTTP 429 and the message "Too many links created from this network."
2. Use case ends in failure.

## Postconditions

### Success Postconditions

- A new `LINK` row exists with a unique `slug`.
- The shortened URL is rendered to the visitor and is on their clipboard.
- For anonymous links: `expiresAt = createdAt + 30 days`. For owned links: `expiresAt = null`.

### Failure Postconditions

- No `LINK` row is created.
- No clipboard write occurs.
- The visitor sees an actionable error message in the form.

## Business Rules

### BR-001: Slug immutability

Slugs are case-sensitive and immutable once issued. They cannot be edited, transferred, or recycled.

### BR-002: Anonymous link expiry

Anonymous links (no `ownerId`) expire 30 days after `createdAt`. Owned links never expire.

### BR-003: Per-IP rate limit

A single IP cannot create more than 20 links per hour. The window is rolling.

### BR-004: Blocklist check is generic

When a destination is on the blocklist, the rejection message must not reveal which list or rule was matched. This prevents probing.

## Non-Functional

- p95 response time < 200ms under 100 RPS (NFR-001).
- Slug uniqueness must hold under concurrent writes via a database-level unique constraint (NFR-003) — never application-level locking.

## Traceability

- **Requirements:** FR-001 (Shorten URL), FR-002 (Custom slug), NFR-001 (Latency), NFR-003 (Slug uniqueness), NFR-004 (Anonymous rate limit), NFR-005 (Anonymous link expiry).
- **Entities:** writes `LINK`; reads `USER` (when authenticated, to set `ownerId`).
