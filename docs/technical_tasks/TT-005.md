# Technical Task: Re-Read Before Write Guard on Gate-File Writes

## Overview

| | |
|---|---|
| **Task ID** | TT-005 |
| **Task Name** | Re-Read Before Write Guard on Gate-File Writes |
| **Category** | Infrastructure |
| **Goal** | Guard against lost updates if gate/log files (`docs/delivery/*-iterations.md`, tracking artifacts) are ever written by more than one agent in the same pipeline run, by documenting a re-read-immediately-before-write convention. Speculative/future-proofing — no known live race today, since delivery is currently sequential. |
| **Status** | Done |

## Acceptance Criteria

- [x] `nexa-claude-core/shared/tracking/TRACKING.md` documents a "Re-Read Before Write" convention for gate/log file writes
- [x] `deliver-use-case/SKILL.md`'s Delivery Log section instructs re-reading the file immediately before appending
- [x] `resolve-bug/SKILL.md`'s Step 2 documents its actual write-target relationship (no shared write target between `bug-analyst` and `requirement-linker` today, so the guard does not apply there — see Decisions)
- [x] `scripts/sync-shared.sh --check` passes after the `TRACKING.md` edit is propagated to the nextjs copy
- [x] No behavior change for current sequential-only usage — this is a guard for future parallel use, not a fix for an active bug

## Affected Areas

- `nexa-claude-core/shared/tracking/TRACKING.md`
- `nexa-claude-nextjs/shared/tracking/TRACKING.md` (synced copy, regenerated via script — never hand-edited)
- `nexa-claude-nextjs/skills/deliver-use-case/SKILL.md`
- `nexa-claude-nextjs/skills/resolve-bug/SKILL.md`

## Dependencies

- None

## Decisions

| Decision | Provenance | Source/Reasoning |
|----------|------------|------------------|
| Extend `TRACKING.md` rather than create a new shared file | INFERRED | `TRACKING.md` is already the closest "how pipelines persist state" doc, already synced via `sync-shared.sh` with no manifest edit needed, and the issue explicitly asks to keep this minimal. |
| Guard is documented convention, not enforced tooling | EXPLICIT | GitHub issue #22 scopes this as a guard/convention, not a locking mechanism — no known live race to justify more. |
| Guard excludes GitHub issue comments and does not apply to `resolve-bug` Step 2's `bug-analyst`/`requirement-linker` pair | INFERRED (corrected via code review) | `code-review` found the original wording described a race that doesn't exist: `bug-analyst` only posts an atomic GitHub comment (no `Edit`/`Write` tool), and `requirement-linker` is the sole writer of the bug doc's Related Artifacts section — the two agents share no write target. Comments are atomic appends with no existing content to lose, so re-read-before-write doesn't apply to them either. |
